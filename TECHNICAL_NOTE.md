# Clawnitizer: Technical Architecture & Design Notes 🛠️🧠

This document outlines the internal mechanics, data parsing strategy, and cross-platform runtime considerations for **Clawnitizer**. It is intended for contributors, system architects, or developers looking to adapt this tool for other agent frameworks.

---

## 🏗️ 1. Core Data Stream Architecture

Clawnitizer operates exclusively on the **JSON Lines (JSONL)** format—specifically targeting the transaction history logs (`session.jsonl`) generated by local autonomous agents. 

### The Dual-File Ecosystem
In standard agent setups (such as `picoclaw` or `openclaw`), a chat session is split into two distinct structures:
1. **`session.meta.json`**: A standard, single-object JSON file containing routing aliases, API keys, and metrics. **Clawnitizer intentionally bypasses this file.** The agent framework dynamically auto-heals or appends data regardless of metadata state indexes.
2. **`session.jsonl`**: A stream of line-delimited JSON string objects. Each line represents a completely encapsulated message turn. This is Clawnitizer's operational target.

---

## 🔄 2. State Preservation & Symmetric Parsing Strategy

To maintain a zero-dependency, single-file footprint, Clawnitizer employs a symmetric read/write pipeline via native PHP. A key design priority is preventing **structural data loss** during manual human edits.

### The Parsing Pipeline
```
[Raw session.jsonl] ──> (Read & Split by Lines) ──> (JSON Decode per Line) │ [Symmetric Swaps] │ [Saved session.jsonl] <── (JSON Encode + Glue) <─── (Render HTML Textareas)
```

### Handling Complex State Components (Hidden Caching)
An AI agent log contains architectural payload parameters crucial for framework execution tracing (e.g., `tool_calls` schemas, function arguments, and explicit tracking UUIDs like `tool_call_id`). 
* **The Problem:** Presenting raw JSON array payload schemas inside a standard textbox ruins the user experience and invites accidental syntax corruption from human editors.
* **The Solution:** Clawnitizer dynamically extracts these tracking arrays during the read phase and caches them as serialized strings inside HTML `<input type="hidden">` nodes. When the user hits **Save**, these structural variables are silently glued back into the newly optimized turn object before string compression.

## 👁️ 3. Whitespace & Character Escape Safety

Standard textareas natively strip or visually hide literal string escape sequences like `\n` (newlines) or escaped quotes. If edited blindly, saving the text field would write corrupted, single-line blocks back to the `.jsonl` stream, causing the agent's internal parser to crash on boot.

### The Anchor Character Swap Mechanics

1. **Ingest Phase:** The engine parses the line array. Every literal `\n` instance inside the message strings is explicitly translated into a visible placeholder sequence: `[NEWLINE]\n`. 
2. **Display Phase:** This forces the browser to render a clear visual marker (`[NEWLINE]`) immediately followed by an actual line wrap inside the editing textbox.
3. **Commit Phase:** Upon POST submission, a strict string substitution (`str_replace`) reverses the process, transforming `[NEWLINE]\n` back into a standardized JSON-escaped `\n` sequence.

## 📈 4. Real-Time Token Calculation Heuristics

Because loading a full tokenizer library (like Tiktoken or SentencePiece) would destroy the single-file, zero-dependency philosophy, Clawnitizer relies on a deterministic **Character-to-Token Ratio Heuristic**.

$$\text{Tokens} \approx \left\lceil \frac{\text{UTF-8 Character Length}}{4} \right\rceil$$

### Dynamic UI Recalculation

* **On Ingest:** PHP computes the byte and character length of each component (`content`, `reasoning_content`, and tool schemas) to render static token badges next to each turn block.
* **On Prune (DOM Modification):** A vanilla JavaScript event listener intercepts the card deletion routine. It pulls the token integer from the target turn's badge, subtracts it from the global header's total state buffer via a simple arithmetic operation, and updates the view live without triggering a server reload.
